# SAP SIM — Backend Bug Report

Generated by the comprehensive pytest API test suite (`test_api_endpoints.py`).
**Date:** 2026-04-14
**Phase:** 7.8

## BUG-001: `DecisionBoard.__init__()` called with non-existent `projects_root` parameter

| Field | Value |
|-------|-------|
| **Severity** | 🔴 Critical (endpoint 500s) |
| **File** | `api/routes.py` → `_get_decision_board()` |
| **Endpoint** | `GET /api/projects/{name}/decisions`, `POST /api/projects/{name}/decisions` |
| **Status** | ✅ Fixed |

**Description:** The helper `_get_decision_board()` passed `projects_root=str(PROJECTS_BASE)` to `DecisionBoard.__init__()`, but `DecisionBoard` only accepts `project_name` and `consensus_threshold`. Every call to the decisions endpoints raised `TypeError`.

**Fix:** Removed the `projects_root` keyword argument.

## BUG-002: `delete_project` does not remove project from SQLite database

| Field | Value |
|-------|-------|
| **Severity** | 🟠 High (data integrity) |
| **File** | `api/routes.py` → `delete_project()` |
| **Endpoint** | `DELETE /api/projects/{name}` |
| **Status** | ✅ Fixed |

**Description:** `delete_project()` only called `shutil.rmtree()` on the filesystem directory but never removed the project record from the SQLite database. After deletion, the project still appeared in `GET /api/projects/{name}` (200 instead of 404), and a second `DELETE` returned 204 instead of 404.

**Fix:** Added `delete_project_state()` helper to `utils/persistence.py` and `delete_project()` method to `db/repository.py`. Route now calls `await delete_project_state(project_name)` before filesystem cleanup.

## BUG-003: `GET /feed` reads from non-existent JSONL file instead of database

| Field | Value |
|-------|-------|
| **Severity** | 🔴 Critical (endpoint always returns empty) |
| **File** | `api/routes.py` → `get_feed()` |
| **Endpoint** | `GET /api/projects/{name}/feed` |
| **Status** | ✅ Fixed |

**Description:** `get_feed()` read events from `feed/events.jsonl` on the filesystem, but `append_feed_event()` (the only write path) stores events in the SQLite database via `db.append_event()`. The JSONL file was never written, so the feed was always empty regardless of how many events existed.

**Fix:** Rewrote `get_feed()` to use new `get_feed_events()` / `count_feed_events()` persistence helpers that query the database directly. Added `count_events()` method to `db/repository.py`.

## BUG-004: `_list_project_names()` scans filesystem instead of querying database

| Field | Value |
|-------|-------|
| **Severity** | 🔴 Critical (endpoint always returns empty list) |
| **File** | `api/routes.py` → `_list_project_names()` |
| **Endpoint** | `GET /api/projects` |
| **Status** | ✅ Fixed |

**Description:** `_list_project_names()` scanned `PROJECTS_BASE` for directories containing `project.json`, but project state is stored exclusively in SQLite (no `project.json` file is written). The list endpoint always returned `[]`.

**Fix:** Replaced filesystem scan with `list_project_names()` persistence helper that queries `db.list_projects()`.

## BUG-005: `DecisionBoard.get_board()` (async) called without `await`

| Field | Value |
|-------|-------|
| **Severity** | 🔴 Critical (endpoint 500s) |
| **File** | `api/routes.py` → `get_decisions()`, `propose_decision()` |
| **Endpoint** | `GET /api/projects/{name}/decisions`, `POST /api/projects/{name}/decisions` |
| **Status** | ✅ Fixed |

**Description:** `DecisionBoard.get_board()` is an async method that queries the database, but the route called it synchronously (via `run_in_executor`), producing a coroutine object instead of actual results. Subsequent iteration over the coroutine raised `TypeError`.

**Fix:** Replaced the entire thread-executor + artifact pattern with direct async DB access via `utils.persistence.get_decisions()` and `utils.persistence.save_decision()`.

| **Endpoint** | `GET /api/projects/{name}/test-strategy` |
| **Status** | ✅ Fixed |

**Description:** Two bugs in one endpoint:

## BUG-010: `DecisionBoard.save()` method does not exist

| Field | Value |
|-------|-------|
| **Severity** | 🔴 Critical (endpoint 500s) |
| **File** | `api/routes.py` → `propose_decision()` |
| **Endpoint** | `POST /api/projects/{name}/decisions` |
| **Status** | ✅ Fixed (eliminated by BUG-005 refactor) |

**Description:** After proposing a decision, the route called `artifact_board.save()`, but `DecisionBoard` has no `save()` method (persistence is done via `save_decision_to_db()`). This would have raised `AttributeError` if the unawaited `get_board()` bug (BUG-005) hadn't already crashed the request.

**Fix:** Replaced the entire propose flow with direct DB access via `utils.persistence.save_decision()`.

## BUG-011: `engine.create_project()` during API project creation overwrites IDLE state

| Field | Value |
|-------|-------|
| **Severity** | 🟠 High (simulation lifecycle broken) |
| **File** | `api/routes.py` → `create_project()` |
| **Endpoint** | `POST /api/projects` → `POST /api/projects/{name}/start` |
| **Status** | ✅ Fixed |

**Description:** The `create_project` API handler called `engine.create_project()` which internally runs `Conductor.initialize_simulation()`. The Conductor transitions the project state from IDLE → RUNNING and saves it to the DB. When the user subsequently calls `POST /start`, `_require_status(allowed=[IDLE, STOPPED])` finds status=RUNNING → 409 Conflict. The simulation could never be started through the normal API flow.

**Fix:** Removed the `engine.create_project()` call from the `create_project` API handler. Engine registration is now deferred to `start_simulation` where it belongs.

**Root cause pattern:** Most bugs (1, 5–10) stem from the API routes being written against an older or assumed artifact API that doesn't match the actual implementations. The DB/filesystem split bugs (2–4) stem from an incomplete migration from file-based to SQLite-based persistence.

**All 11 bugs have been fixed.** The test suite (`80 tests`) validates every endpoint and all pass.
